# Contributing to `flepimop2-extras`

This document provides guidelines and instructions for contributing to
`flepimop2-extras`, including the repository organization, development
conventions, and tips for best practices.

## Repository Organization

`flepimop2-extras` is a collection of external provider packages for
[`flepimop2`](https://github.com/ACCIDDA/flepimop2), with one package per
module. It is structured as a [`uv` workspace](https://docs.astral.sh/uv/concepts/projects/workspaces/):
the repository root is a non-distributable workspace coordinator that holds the
shared development tooling, while each distributable package lives under
`packages/`.

```
flepimop2-extras/
|-- pyproject.toml                # Workspace root: shared ruff/mypy/pytest config + dev deps
|-- uv.lock                       # Single lockfile for the whole workspace
|-- justfile                      # Command runner recipes for every package
|-- LICENSE                       # GPLv3, shared by every package (single source of truth)
|-- AUTHORS                       # Shared by every package (single source of truth)
|-- .yamllint.yaml
|-- .github/                      # CI workflows, composite actions, dependabot
|-- packages/
    |-- flepimop2-{feature}/      # One package per provider module
        |-- pyproject.toml        # Package metadata + dependencies only
        |-- README.md
        |-- LICENSE               # Generated by `just sync-licenses`; gitignored
        |-- AUTHORS               # Generated by `just sync-licenses`; gitignored
        |-- src/
        |   |-- flepimop2/        # NO __init__.py (namespace package)
        |       |-- {category}/   # NO __init__.py (namespace package)
        |           |-- {name}/   # HAS __init__.py (the actual module)
        |               |-- __init__.py
        |               |-- py.typed
        |-- tests/
```

### Packages

Each package contributes a module to the `flepimop2.*` namespace following
[PEP 420](https://peps.python.org/pep-0420/) implicit namespace packages, so it
can be installed alongside `flepimop2` core and other provider packages without
conflicts. See the upstream guide on
[creating an external provider package](https://accidda.github.io/flepimop2/latest/development/creating-an-external-provider-package/)
for the full details.

The conventions every package must follow:

- **Package name**: `flepimop2-{feature}` (e.g. `flepimop2-slurm`).
- **Source layout**: `src/flepimop2/{category}/{name}/`. Do **not** create
  `__init__.py` files in `src/flepimop2/` or `src/flepimop2/{category}/` - these
  directories must remain empty to function as namespace packages. Only the leaf
  module directory gets an `__init__.py`.
- **Build config**: each package's `pyproject.toml` sets
  `[tool.hatch.build.targets.wheel] packages = ["src/flepimop2"]` so Hatch
  packages everything under `src/flepimop2` as part of the `flepimop2` namespace.
- **Typing**: ship a `py.typed` marker so downstream type checkers pick up the
  package's annotations.
- **License**: external provider packages must be licensed under GPLv3 to comply
  with `flepimop2`'s license. The `LICENSE` and `AUTHORS` files are centralized
  at the repository root as the single source of truth; `just sync-licenses`
  copies them into each package (the copies are gitignored) so they ship in the
  built artifacts. Declare `license = "GPL-3.0-or-later"` in the package's
  `pyproject.toml` and add the standard GPLv3 header at the top of each source
  and test file.
- **Version & flepimop2 constraint**: every package is versioned in tandem and
  shares the same `flepimop2` dependency constraint. Both are defined **once**
  at the repository root under `[tool.flepimop2-extras]` (`version` and
  `flepimop2-constraint`) and propagated into each package's `[project].version`
  and `flepimop2` dependency by `just sync-versions`. Do not hand-edit a
  package's version or `flepimop2` constraint - change the root values and run
  `just sync-versions`. `just check-versions` (part of `just quality`, and run
  in CI) fails if any package has drifted.
- **Python**: target `requires-python = ">=3.11,<3.15"`.

Shared tooling (ruff, mypy, pytest, coverage) is configured **once** at the
workspace root in `pyproject.toml`. Packages should not redefine this
configuration.

The repository root is a metapackage: `flepimop2-extras` ships no code of its
own but exposes each provider package as an optional dependency, so users can
run `pip install 'flepimop2-extras[slurm]'` to pull in `flepimop2-slurm`, or
`pip install 'flepimop2-extras[all]'` to install everything. (`tool.uv.sources`
is a development-only override that uv strips from the published artifact, so
the metapackage depends on the packages from PyPI.)

### Adding a New Package

1. Create `packages/flepimop2-{feature}/` following the layout above. For the
   package's `version` and `flepimop2` dependency, just use any placeholder -
   they will be overwritten in step 3.
2. Wire the package into the root `pyproject.toml`:
   - `[tool.uv.sources]` as a `{ workspace = true }` source, and the `dev`
     dependency group, so a single `uv sync` installs the whole workspace.
   - `[project.optional-dependencies]`: add a `{feature}` extra for the package
     and append it to the `all` extra, so it is installable via the metapackage.
3. Run `just sync-versions` to set the package's version and `flepimop2`
   constraint from the root `[tool.flepimop2-extras]`, then `just sync-licenses`
   to populate its `LICENSE` and `AUTHORS` (both also run automatically as part
   of `just build-check`).
4. Run `uv sync` to update `uv.lock`, then `just` to verify everything passes.

## Development Setup

### Prerequisites

- Python 3.11, 3.12, 3.13, or 3.14.
- [`uv`](https://docs.astral.sh/uv/) - Python package manager.
- [`just`](https://just.systems/) - Command runner (optional but strongly
  recommended).

### Initial Setup

1. Clone the repository:

```shell
git clone git@github.com:ACCIDDA/flepimop2-extras.git
cd flepimop2-extras
```

2. Create a virtual environment and install dependencies:

```shell
uv sync
```

This creates a `.venv` virtual environment and installs every workspace package
along with the shared development dependencies (mypy, pytest, ruff, yamllint).
If you need to create an environment with a specific Python version you can also
run:

```shell
uv sync --python 3.12
```

3. Verify your setup:

```shell
just
```

This runs the default development checks:

- `just ruff` - Format code and run lint checks with auto-fixes.
- `just mypy` - Type check the codebase.
- `just test` - Run tests with coverage.
- `just yamllint` - Lint YAML files.

It is recommended that you run this command frequently as you do development
work to catch issues early. You can also run `just --list` to list out the
available commands.

## Code Standards

### Style

We use [`ruff`](https://docs.astral.sh/ruff/) for both formatting and linting:

- **Formatting**: `ruff format` follows the Black code style.
- **Linting**: `ruff check` enforces code quality rules.

Run `just` to automatically format/lint, type check, and run tests before
committing. If you only want formatting/linting, run `just ruff`.

### Docstring Style

[Google style](https://google.github.io/styleguide/pyguide.html#s3.8-comments-and-docstrings)
docstrings are used to document code.

## Testing

### Organization

Each package keeps its tests in its own `packages/flepimop2-{feature}/tests/`
directory, mirroring the source code structure:

```
packages/flepimop2-{feature}/tests/
|-- {module}/
|   |-- test_{function}.py                   # Tests for individual functions
|   |-- test_{class_in_snake_case}_class.py  # Tests for classes
|-- test_{function}.py
```

**Examples**

- For a function called `flepimop2.job.slurm.foo` the unit test file would be
  located at `packages/flepimop2-slurm/tests/test_foo.py`.
- For a class called `flepimop2.job.slurm.SlurmJob` the unit test file would be
  located at `packages/flepimop2-slurm/tests/test_slurm_job.py`.

### Running Tests

Test execution is exposed through `just`:

- `just test` - Local default test workflow (runs `just cov`).
- `just cov` - Run the test suite, report coverage, and enforce the configured
  minimum coverage threshold.
- `just pytest` - Run the full pytest suite without coverage reporting.

Unlike `flepimop2` core, this repository has no integration-test split - these
modules have a smaller surface area, so unit tests (and doctests) are
sufficient.

For more advanced test runs, use `pytest` directly:

```shell
uv run pytest packages/flepimop2-{feature}/ -v               # Run all tests in a package
uv run pytest packages/.../tests/test_{function}.py -v       # Run a specific test file
```

For more information on how to invoke pytest please refer to the
[How to invoke pytest](https://docs.pytest.org/en/stable/how-to/usage.html)
documentation.

### Writing Tests

- Any public API should have unit tests that reaffirm the documentation's
  description.
- If possible unit tests should use `@pytest.mark.parametrize` for generality
  and ease of adding new test cases.
- Use descriptive test names that explain what is being tested. In the case of
  testing exceptions also the type of exception.
- For smaller helper functions, especially internal helpers, doctests are
  sufficient.

### Type Checking

All code must pass strict type checking with `mypy` which can be invoked with
`just mypy`. Note that `ruff` will catch missing type hints whereas `mypy` will
check that those type hints are correct and consistent.

## Packaging

Packaging checks are available through the `build` recipe group:

- `just build-check` - Build the source distribution and wheel for every
  package, then run `twine check --strict` on the generated artifacts.

This command is useful when preparing a release or when making changes that
could affect packaging metadata, installability, or build artifacts.

## Pull Request Process

### Before Submitting

1. Run CI checks locally using `just ci`. This runs the same checks that CI
   will run:

- `just quality` - CI quality checks (`ruff` check mode + `mypy`).
- `just test` - Coverage-gated tests.

In GitHub Actions, these correspond to the `quality` and `tests` jobs in
`.github/workflows/ci.yaml`. The separate `package` job runs `just build-check`,
and a non-blocking `flepimop2-dev` job runs the test suite against the
development (`main` branch) version of `flepimop2` as an early heads-up for
upcoming breakage.

If you edit YAML files, also run `just yamllint`. YAML linting runs as its own
`yaml` workflow in CI.

2. Add tests for new functionality or bug fixes. Particularly for bug fixes, the
   test should be written before the fix and fail without the fix present.

3. Update the relevant package's `README.md` if your changes affect user-facing
   functionality.

### Submitting a Pull Request

1. Create a branch from `main` and make your changes on this branch using the
   code standards above. Please use clear and descriptive commit messages that
   explain the changes made and why. After your work is finished either push
   directly to `flepimop2-extras` or your fork (depending on your permissions).

2. Create a pull request with:

- The motivation for and a clear description of the changes.
- Link any related issues (use "Closes #XYZ" to auto-close issues).

### Pull Request Requirements

- Tests run in CI against Python 3.11, 3.12, 3.13, and 3.14 using the committed
  `uv.lock`.
- Quality checks run in CI on Python 3.12.
- Package validation runs in CI on Python 3.12.
- At least one maintainer approval is required before merging.
- Branches must be up to date against `main` before merging and have a linear
  history. Only rebases are allowed for merging.

## Reporting Issues

### Bug Reports

When reporting bugs, please include:

- Operating system and version.
- Python version (e.g., Python 3.12.1).
- The affected package and its version as a git commit hash.
- A minimal, reproducible example that demonstrates the issue.
- An explanation of expected behavior vs actual behavior.
- Error messages or tracebacks if applicable.

### Feature Requests

For feature requests, please:

- Check existing issues to avoid duplicates.
- Clearly describe the use case and motivation.
- Provide examples of how the feature would be used.
- Be open to discussion about implementation approaches.

### Questions

For questions about using `flepimop2-extras`:

- Search existing issues for similar questions.
- Open a new issue with the "question" label.
